### ⚠️ Pre-Mortem 立刻命中的两个「架构换代」事实(README 看不到,是命门)

1. **Qwen3.5-2B 不是普通 dense transformer,而是「Gated DeltaNet(GDN,线性注意力 / RNN 风格)+ Gated Attention」混合架构**——约 75% 层是 GDN 线性注意力,25% 是全注意力(3:1 layout,继承 Qwen3-Next)。这与 Qwen3-1.7B 的标准稠密 transformer **根上不同**。
   来源: [HF blog mlabonne "Qwen3.5: Nobody Agrees on Attention Anymore"](https://huggingface.co/blog/mlabonne/qwen35) · [Spheron deploy Qwen3.5 GDN guide](https://www.spheron.network/blog/deploy-qwen-3-5-gpu-cloud/) · [gist qwen3.5-gated-deltanet-analysis](https://gist.github.com/justinchuby/0213aa253664fb72e9adb0089816de15)

2. **Qwen3.5-2B 是「统一 VLM」(一套权重含 vision encoder),不是独立文本模型**。可在推理时关 vision 跑纯文本(`--language-model-only` / `mlx-lm` 文本路径),但权重/依赖带视觉成分。这与项目当前「纯文本 Qwen3-1.7B」不同。
   来源: [HF Qwen/Qwen3.5-2B card](https://huggingface.co/Qwen/Qwen3.5-2B) · [artificialanalysis](https://artificialanalysis.ai/articles/qwen3-5-small-models)(2B 在 MMMU-Pro 多模态得 25.8%)· [unsloth: "Causal LM with Vision Encoder"](https://unsloth.ai/docs/models/qwen3.5)

> 这两点决定了:升级不是「换个更大的同款」,而是「换架构 + 换模态形态」,故定性为**条件升级**而非「直接升」。

## 1. 核心问题逐条回答

### ① Qwen3.5-2B 比 1.7B 强多少?(tool-call / 中文 / 泛化)

| 维度 | Qwen3-1.7B(基线) | Qwen3.5-2B(候选) | 提升 | 证据强度 |
|---|---|---|---|---|
| **本地 judgment tool-call**(tool-calling-benchmark Agent Score,20-run,CPU) | **0.960(sub-2B 第一)** Action 0.900 / Restraint 1.000 / Wrong-Tool 0 | **无实测**(该 bench 未含任何 Qwen3.5) | 未知 | 🔴 基线强,候选**未证实** |
| **BFCL overall** | ~55.5%(TinyLLM 论文,多轮仅 ~16.9%) | **无 2B 实测**(BFCL 只有 122B/397B 旗舰分:122B-A10B=72.2%) | 未知 | 🔴 候选 2B **未证实** |
| **综合 Intelligence Index**(artificialanalysis) | ~13(估) | **16**(+3) | +3 分,约 +23% | 🟡 复合指标,非 tool-call 专项 |
| **中文 C-Eval / CMMLU** | C-Eval 66.7 / CMMLU 66.6(三方报告) | **无 2B 官方中文表** | 未知 | 🟡 1.7B 有第三方分,候选未证实 |
| **官方定性 tool-call** | 强(Qwen3 native) | 「Qwen3.5 excels in tool calling」「Qwen-Agent / MCP」 | 官方说强,无 2B 数字 | 🟡 定性,无量化 |

**结论(诚实)**: 唯一可量化的代际提升是 **Intelligence Index +3 分(~23%)**——「meaningful uplift」但**不是数量级跃迁**;artificialanalysis 称 Qwen3.5-2B 综合智力「匹配 Falcon-H1R-7B、超 Nemotron Nano 9B」(若属实,小尺寸下很可观)。**但车控单跳 tool-call / 中文 NLU / restraint 这三个 demo 命门维度,Qwen3.5-2B 没有任何公开实测分,全部「未证实」**。⚠️ 关键警告: 当前 1.7B 在本地 judgment 基准上是 sub-2B 冠军(0.960),升级到一个「这些维度未实测」的新架构模型,**有回退风险**(更大 ≠ 更会克制,benchmark README 反复证明参数量是 weak predictor: 0.6B 0.880 > 1.7B 0.670 曾出现过)。
来源: [tool-calling-benchmark README(本地)](file:///Users/wanglei/workspace/raw/05-Projects/MAformac/ref-repos/tool-calling-benchmark/README.md) · [arxiv TinyLLM 2511.22138](https://arxiv.org/pdf/2511.22138) · [artificialanalysis Qwen3.5 small](https://artificialanalysis.ai/articles/qwen3-5-small-models) · [Qwen3 技术报告 C-Eval/CMMLU 三方值]

### ② iPhone 端量化后能跑吗?(RAM / 延迟 / 包体)

**能跑,且 decode 反而更快。** 在 4B 天花板(D38)之下,2B 有充足余量。

| 指标 | Qwen3-1.7B(当前) | Qwen3.5-2B(候选) | 来源 |
|---|---|---|---|
| iOS decode | ~42 tok/s(MLX-Swift,Release) | **61.2 tok/s**(iPhone 17 Pro,MLX-Swift,4bit) | 1.7B: `wizcheu/iOSLLMFrameworkBenchmark`+ qwen3-engineering-notes;2B: [john-rocky/apple-silicon-llm-bench](https://github.com/john-rocky/apple-silicon-llm-bench) |
| 峰值 RAM | ~1.4GB | **~1.28GB**(1,279MB,iPhone 17 Pro) | 同上 |
| load | 0.68s | 未单列 | qwen3-engineering-notes:71 |
| CoreML/ANE 路径 | — | 峰值内存比 MLX 少 3-8×(decode ~半速) | apple-silicon-llm-bench |

⚠️ **数据可信度警告**: apple-silicon-llm-bench 那一行标 "Qwen 3.5 2B" **但同仓 VLM 相机段引用的是 `mlx-community/Qwen3-VL-2B-Instruct-4bit`(旧 Qwen3-VL 线,不是新 GDN 的 `Qwen/Qwen3.5-2B`)**。WebFetch 校验显示「标准 2B 行似指文本变体」但**未能 100% 确证是 GDN 版**。故 61.2 tok/s **可能高估了 GDN 版 decode**(GDN 的 RNN 状态更新在 Metal 上若无优化 kernel 会更慢)。**端侧 RAM/包体不是 blocker(2B 4bit ≈ 1.3-1.5GB,iPhone 12+ 8GB 内存够),但 tok/s 必须自测确认。**

### ③ MLX-swift / llama.cpp 对 Qwen3.5-2B + qwen3_coder 格式可用吗?(命门)

**这是最大的 tiger。** 三个互锁问题:

**(a) GDN 架构 runtime 支持**:
- **mlx-lm(Python)**: 已合/正合入 GDN 支持,`mlx-community` 有 4bit 转换权重。
- **mlx-swift(核心 Swift API)**: **❌ 主线未见原生 GDN 线性注意力层**(内置 recurrent 仅 RNN/LSTM/GRU + MultiHeadAttention,无 Gated DeltaNet 层)。GDN 在 Swift 生态目前靠**第三方**实现(Soniqo Qwen3Chat / Rapid-MLX / SharpAI SwiftLM)。
- **llama.cpp**: 支持但需最新 build——旧 GGUF 转换会触发 `FGDN_AR` tensor-prefix 断言崩溃([llama.cpp #23347](https://github.com/ggml-org/llama.cpp/issues/23347))。
  来源: [ml-explore/mlx-swift](https://github.com/ml-explore/mlx-swift) · [Soniqo docs](https://soniqo.audio/guides/chat) · [Rapid-MLX](https://github.com/raullenchai/Rapid-MLX)

**(b) tool-call 格式不兼容(🔴🔴 命门中的命门,且与项目已有 pre-mortem 直接撞车)**:
- **stock Qwen3.5 chat template 不发 `<tool_call><function=...>` 标签**,而 `qwen3_coder` parser 只在模板含这些标签时激活——`_infer_tool_parser()` 返回 `None` → **不解析工具 → 模型返回 `content:""` 无 tool_calls(静默失败)**。
  来源: [mlx-lm #1293 "add tool parser support for Qwen 3.5/3.6 (non-Coder)"](https://github.com/ml-explore/mlx-lm/issues/1293) · [QwenLM/Qwen3 #1831 fix chat template](https://github.com/QwenLM/Qwen3/issues/1831) · [froggeric Qwen-Fixed-Chat-Templates](https://huggingface.co/froggeric/Qwen-Fixed-Chat-Templates)
- **这正是项目自己 pre-mortem 早记的 T2**(`docs/execution-pre-mortem-2026-06-18.md:12`): 「`mlx-swift-lm` `.json` parser 只认 `<tool_call>` 标签,裸 JSON 当文本透传不 fallback」,且 `intent-routing-explore` 实测 base 1.7B 已有 22.5% 因「裸 JSON 无标签」走错通道。**升 Qwen3.5-2B 会把这个已知坑放大**——3.5 的默认模板本就不发标签。
- **修法已知**: 用 froggeric 修复模板(恢复 Coder XML 标签)或换 `--tool-call-parser qwen3_xml`。但 **#1293 是针对 `mlx-lm`(Python),`mlx-swift-lm`(项目主 backend)是否有等价 `_infer_tool_parser` 逻辑未核实**——必须直读 Swift 源码验证。

**(c) thinking-loop bug(2B 特有)**: 官方 card 明示「Qwen3.5-2B 比其它 Qwen3.5 更易进入 thinking loop,可能不终止生成」,建议调采样参数 + 流式中断。对「秒回不崩」的 demo 北极星是直接威胁。
  来源: [HF Qwen/Qwen3.5-2B card](https://huggingface.co/Qwen/Qwen3.5-2B)

### ④ LoRA 训练 2B vs 1.7B 成本

| 项 | Qwen3-1.7B | Qwen3.5-2B |
|---|---|---|
| unsloth 支持 | ✅ 成熟(多个现成脚本: strands-mlx / Qwen3-1.7b_finetune) | ✅ 支持(`Fine-tune Qwen3.5` 官方 guide + Colab) |
| LoRA VRAM | ~4-5GB(估) | **~5GB**(明示,GTX 1060 级) |
| 额外依赖 | 无 | **需 vision 依赖(torchvision/pillow,因是 VLM)** |
| 架构成熟度 | 高 | ⚠️ 社区初期吐槽「pretty weird with the delta nets」([unsloth #4108](https://github.com/unslothai/unsloth/issues/4108)),现已支持 |
| LoRA 友好性 | 标准 | ✅ control tokens 训练支持 PEFT,无需训 embedding(利好) |

**结论**: LoRA 成本**几乎不增**(~5GB,消费级 GPU 可跑),不是 blocker。唯一注意: VLM 依赖 + DeltaNet 早期实现需 transformers/unsloth 最新版。
来源: [unsloth Qwen3.5 fine-tune docs](https://unsloth.ai/docs/models/qwen3.5/fine-tune) · [Medium "5GB VRAM"](https://medium.com/coding-nexus/fine-tune-qwen3-5-on-your-own-gpu-you-only-need-5gb-vram-1f8e46f15631) · [apxml Qwen3.5-2B VRAM](https://apxml.com/models/qwen35-2b)

### ⑤ 不降级判定: 升 or 退?

**判定: 条件升级(不盲升,不轻退)。** 端侧硬件可行(RAM/包体/decode 都过),提升方向存在(Intelligence Index +3)但**未在 demo 命门维度(tool-call/中文/restraint)证实**,且引入「GDN runtime + tool-parser + thinking-loop」三个新风险面。**不满足「直接升主力」,但也远未到「真跑不动该退」**。正确动作 = **先 spike 验 3 个死门,通过则升主力(1.7B fallback),不通过则维持 1.7B 主力**。

## 5. 给磊哥的拍板建议

⭐ **推荐: 条件升级(`需spike验证`)。** 量化预算:
- **S1+S2 两个死门 spike ≈ 0.5-1 天**(直读 mlx-swift-lm 源码 + iPhone 真机实采 tool-call & TTFT)。这是「升不升」的决策门,**先做这个,别先训 LoRA**。
- S1 通过 + S2 通过 → **升 Qwen3.5-2B 主力,1.7B 降 fallback**(decode 更快、RAM 更低、智力 +3,且端侧有余量)。
- S1 **不**通过(parser 硬墙) → **维持 1.7B 主力**(它是本地 judgment 冠军,已实装,零迁移成本),记录 Qwen3.5 待 mlx-swift 原生支持成熟再评。
- 不要因为「2B 更新更大」就盲升: 当前 1.7B 在最贴 demo 的 judgment tool-call 基准上是 sub-2B 冠军(0.960),升级的唯一正当理由是 spike 证明 2B 在**这些维度不回退**且端侧不退化。

**一句话**: 不是「不用 2B」,而是「2B 真实存在、端侧能跑、方向更强,但要先用两个 spike 钉死 tool-parser 兼容 + GDN 端侧延迟这两根命门,通过即升,不通过即守 1.7B」。
