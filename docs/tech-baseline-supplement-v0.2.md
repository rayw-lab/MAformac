## §13 多阶规划层（multi-stage planning）

> 衔接：v0.1 §7 只讲了「快/慢二路由 + 慢思考 ReAct ≤3 轮」与「三态推荐」，把"慢"当成一个扁平的 ReAct 循环。本节把"慢"展开成**可判定边界、有阶段状态机、有显式失败出口、可端侧降级**的多阶规划层，是 v0.1 §7.1 慢路径的下钻细化，不重复三态推荐链（那在 v0.1 §7.2）。

### 13.1 判定边界：什么任务才该走"多阶串行规划"

v0.1 §7.1 给的是「快 vs 慢」二分。本节进一步把"慢"再切成两档：**单轮 ReAct（≤3 跳浅链）** vs **多阶串行规划（plan→execute→observe→re-plan）**，避免所有"慢"都被当成长链路打，浪费 0.6B 算力与延迟预算。

入口先答四问（顺序不可反，错一档后面全在错前提上展开）：是否需要进执行世界 → 是否高风险 → 槽位是否齐全 → 是否存在可补全路径。三档分流：

| 档 | 触发条件 | 去向 | 对应 v0.1 |
|---|---|---|---|
| 单步 FC | `needs_action=true` ∧ `slot_ready=true`（对象/槽位/目标态都清楚）| 直接工具映射，1 跳 | v0.1 §7.1「走快」 |
| 单轮 ReAct（≤3）| 需 1-2 次"查→执行"，链路浅 | 思考→行动→观察 1 循环 | v0.1 §7.1「走慢」浅链 |
| 多阶串行规划 | 见下硬条件全满足 | plan→execute→observe→re-plan | 本节新增（v0.1 未细分）|

升级到多阶的硬条件：`needs_action=true` ∧ `slot_ready=false` ∧ `repair_path=true`（存在稳定补全路径）∧ `safety_gate` 可控。

门槛原则：**复杂意图 + 工具依赖 + 上下文依赖**三项至少满足两项才进多阶。单一明确指令、即时记忆（"上一首"）、垂域精确技能（音乐/影视）一律不进多阶。`repair_path=false`（只有情绪没有可执行锚点）→ 澄清/拒识/降级文本，**非技能严禁进入执行链**（与 v0.1 §11.3 badcase「关怀话术代替执行」互补：那里讲"自由说必须落候选或拒识"，这里讲"无锚点不得进多阶执行"）。

### 13.2 阶段状态机（最小可验证决策循环）

四阶段，每阶段都有**显式失败出口**（v0.1 §9 的 barge-in 状态机管的是全双工打断，本状态机管的是规划内部流转，二者正交叠加：取消令牌仍按 v0.1 §9.2 穿透）：

| 阶段 | 核心问题 | 输出 | 失败处理 |
|---|---|---|---|
| ① 任务成形 | 这句话值不值得进多阶 | 任务对象/目标/约束/候选链路 | 退回 FC / 单轮 ReAct / 文本 |
| ② 规划生成 Planner | 第一步先做什么 | 步骤序列 + 工具参数草案 | 改写 Query 或发起澄清 |
| ③ 行动执行 Tool Router | 真正调了什么 | 工具结果 + 状态码 + 证据 | 重试 / 改参 / 切工具 |
| ④ 观察终止 Stop Gate | 是否完成 | 总结回复 / 下一跳 / 回退 | 输出中止说明 + 兜底 |

四段式运行时对象（immutable，与 v0.1 §7.2 三态引擎纯函数风格一致，每跳返回新对象不就地改）：

```jsonc
TaskContext   { goal, constraints, domain_hint, safety_level }
PlanState     { current_step, remaining_budget, tool_trace[] }
Observation   { result, confidence, latency_ms, retryable }
DecisionGate  { finish, fallback, clarify, handoff }
```

**Stop Gate 必须显式**——多阶最大风险是"停不下来"。终止条件（命中任一即停）：达成目标 / 跳数预算上限 / 命中高风险（移交 v0.1 §4 L4 安全层 / 本篇 §16）/ 证据冲突或槽位缺失 / 用户打断（v0.1 §9 barge-in）。

三类观察值分清职责：**事实观察**（补证据，回 ③）/ **环境观察**（决定动作可执行性，调 v0.1 §7.2 三态判定）/ **过程观察**（决定下一跳或中止，进 ④）。

### 13.3 端侧 0.6B 适配（能做几阶 / 何时降级）

云端原方案是大模型 MasterAgent、最大 3 跳、端到端 ≤3500ms。0.6B 算力/上下文都更紧。**MAformac 端侧多阶上限建议 1–2 阶**（不是 v0.1 §7.1 慢路径笼统的 ≤3 轮——那是单轮 ReAct 的轮数；多阶的"阶"比"轮"重，每阶含一次完整 plan，故收得更紧）。

五级降级链（0.6B 适配版，与 v0.1 §2 降维表、§7 快慢路由对齐，任一步失败/超时自动下沉）：

```
L0 缓存直答
  → L1 单步 FC / mock 车控（v0.1 §5.3 两个核心 mock 工具）
  → L2 垂域 mock 技能（v0.1 §5.1 八大垂域）
  → L3 多阶（0.6B+LoRA，≤2 阶 ← 本节）
  → L4 单步兜底
  → L5 文本闲聊
```

预算硬上限（写进 PlanState.remaining_budget，代码强校验非 prompt 软约束，承接 v0.1 §10 Codex 铁律④）：

- `max_steps ≤ 2`；工具白名单 schema 校验（防 0.6B 工具幻觉，对接 v0.1 §6 四级协议槽值域）。
- 单工具超时即降级；重试 ≤ 1 次（与 v0.1 §9.2「模型修复最多重试 1 次」同源）；整体推理超时即降级。

何时降级到端侧低阶：链路 ≥ 3 跳 / 证据冲突 / 上下文超 0.6B 窗口（注意与 v0.1 §8.1 recent_turns=2 轮窗口约束叠加）/ 置信度低于阈值 → 退回单步 FC 或"已执行 + 简短解释"。

CoT 节流：0.6B 推理慢，快思考模式静默规划不展示思维链；演示需要透明度时才用慢思考逐步播报。**reasoning_steps 作为输出首字段（推理前置）** 提升工具调用准确率、减少重试——这是端侧小模型 FC 的关键工程技巧，v0.1 未提。

### 13.4 多阶演示价值锚点

这正是某车厂的差异化窗口——竞品普遍不支持"模糊话术 → 多车控联动推理"。四个可演示资产（与 v0.1 §11.1 金句表互补：那是单句高光，这是多阶过程高光）：

1. **生理感受 → 多动作联动**：「屁股特别烫」→ 读 mock 端状态 → 调空调+开座椅通风 → 三段式回复"已执行 + 为什么 + 可微调"。
2. **多阶搜索 → 执行**：「调出星巴克品牌色的氛围灯」→ mock 搜索品牌色 → 映射 RGB → mock 车控。
3. **慢思考可视化**：macOS 大屏展示 `plan→execute→observe→re-plan` 状态机流转 + trace 字段，把"多阶在想什么"做成可演示资产。
4. **降级演示**：故意给不可补全输入（`repair_path=false`）→ 展示优雅澄清/回退，证明系统"能停下来"而非死循环。

多阶 trace 日志字段（macOS 大屏可视化用，是 v0.1 §11 eval 字段的运行时补充）：
`trace_id, plan_depth, tool_count, fallback_reason, unsafe_block, completion_mode{完成|澄清|回退|放弃}`。

## §14 中枢调度 + Agentic-Skill 分工（多 domain 编排母版）

> 衔接：v0.1 §4.2 给了 `Registry.route()` / `cbm_tool_pk` 落域仲裁与 Capability 协议，但只到"单层路由打分 → 派发"。本节补三层调用链、角色职责边界、Skill 选型（怎么想）、多技能竞价仲裁四锁，并映射回 v0.1 §4 的 Router。**不重复** Capability schema（v0.1 §4.2）与 domain 清单（v0.1 §5.2）。

### 14.1 三层调用链 + 调用边界

```
用户语音/文本
  → 基础智能体（认知层：理解·生成·Persona·记忆）       对用户可见
  → 中枢智能体（调度层：路由·仲裁·优先级·拒识防护）     幕后  ← v0.1 §4.2 Registry.route() 的上位语义
  → MasterAgent + 垂域 Agent 群（执行层：FC + 端状态推理）执行反馈可见
```

核心语义澄清（v0.1 隐含但未点破）：**MasterAgent 不是中枢，是中枢路由的下游专域执行体之一**（车控域）。三者是「调度者 → 专家」，非并列。大模型是认知能力提供者，不直接触达用户。

### 14.2 三角色职责矩阵 + Agent / Skill 区分

| 角色 | 定位 | 职责 | 不做 |
|---|---|---|---|
| 中枢智能体 | Orchestrator / Router | 意图落域、Skill 选型、跨域协调、冲突仲裁、拒识防护 | 不做语义生成、不直接执行 |
| MasterAgent | 专域执行体（车控）| 结合端状态的多阶推理 + FC 之上的快/慢思考 | 不做路由、不管其他域 |
| Agentic-Skill | 中枢内部的推理模式 | ReAct / Plan-and-Execute / Multi-Agent-DAG，按场景选一 | 不是 Agent，是「怎么想」的策略 |

关键区分（v0.1 未明确，易混）：**Agent = 谁来做（导航/音乐/车控），Skill = 怎么想（ReAct / P&E / DAG）。** 中枢先选 Skill，Skill 内部再激活若干 Agent。本区分与本篇 §13 多阶规划衔接：§13 的「单轮 ReAct vs 多阶」正是 Skill 选型的端侧落地。

### 14.3 落域 → 分发 → 执行 → 回收 完整链路

两层分发开关（Skill 选型前的预判，是 v0.1 §7.1 快慢路由的"上位中枢视角"）：

| 阶段 | 条件 | 路由 |
|---|---|---|
| 0 安全 | 紧急/告警 | 硬实时 Quick Path，绕过 Skill（对接本篇 §16 R3）|
| 1 简单 | 历史熟悉 + 单域单步 | Quick Path 端侧 SLM（= v0.1 §7.1 走快）|
| 2 复杂 | 跨域/多步/需实时数据 | Smart Path → Skill 选型 |

三维升级判断（任一 YES 升 Smart Path）：历史熟悉度低 / 需多 Agent 跨域 / 需实时数据。

Skill 选型分发：

- 模糊感受、需端状态动态调整 → **ReAct**（Thought→Act→Observe，Token 线性增长，端侧 ≤5 轮防截断）
- 长链路确定多步、步骤有依赖 → **Plan-and-Execute**（Planner 一次规划 + Executor 逐步执行 + Re-planner 偏差兜底，Token 不随轮次增长）
- 跨域无依赖并行 → **Multi-Agent / DAG**（Planner 生成 Task-DAG + 并行分派 + Joiner 汇聚仲裁）

回收：各 Agent 结果汇入 Joiner/仲裁层 → 按权重投票 → 主频道（语音）+ 辅助卡片（屏幕弱展示）双通道输出（与 v0.1 §5.3「GUI=事实 / VUI=操作对象」回复分层一致）。

### 14.4 多技能竞价 / 仲裁 / 锁域机制

四层冲突解决（优先级降序，扩展 v0.1 §4.2 单一 pk_score）：

1. 权重优先级：安全 > 导航 > 娱乐 > 舒适
2. 时间窗口：高速行驶禁用娱乐类建议
3. 用户偏好：历史学习权衡（接本篇 §16 L3 记忆）
4. 显式仲裁：无法解决时询问用户

DAG 调度四条工程锁：

- 硬实时插队：安全 Task 插 DAG 头部，不等非安全完成
- Agent 超时熔断：超本级延迟上限返回降级结果（接本篇 §13.3 预算上限）
- 资源互斥锁：竞争同一硬件总线（如空调/车窗同总线）的 Task 串行
- Joiner 投票：冲突建议按权重表决

并行 FC 三前提（缺一退串行）：无数据依赖、无资源竞争、有幂等保证（幂等承接 v0.1 §9.2 执行层幂等可回滚）。

### 14.5 映射成 MAformac 的 Router + Capability 编排

把上面语义压成两组件，**Router 是 v0.1 §4.2 `Registry.route()` 的增强版**（v0.1 只输出 pk → dispatch，这里加 skill 选型 + 拒识）：

```jsonc
RouteResult {
  domain: enum[car_control, navigation, music, chat],     // 落域（= v0.1 §6 一级 service）
  skill:  enum[quick, react, plan_execute, multi_agent],  // Skill 选型 ← 本节新增
  capabilities: [cap_id...],                              // 命中能力（v0.1 §4.2 Capability.id）
  priority: int,                                          // 仲裁权重 ← 本节新增
  reject: bool                                            // 拒识防护 ← 本节新增
}
```

- 0.6B 模型只承担「落域 + Skill 标签 + 拒识」三件事（与本篇 §15 LoRA 多任务头对应）；多阶推理交给 Plan-and-Execute 的 Executor 路径（仍端侧小模型跑）。
- 离线约束：无云端 Smart Path，则"复杂"场景退化为端侧 Plan-and-Execute（一次规划 + 逐步执行），不依赖云大模型（守 v0.1 §1 北极星「断网也能跑」）。

Capability 注册表增量字段（在 v0.1 §4.2 Capability 协议基础上加仲裁/并行所需字段，**不重复** v0.1 已有的 id/domains/intents/connector）：

```jsonc
Capability += {
  weight: int,            // 仲裁权重（§14.4 第 1 层）
  exclusive_bus: str|null,// 互斥锁键（同键 → 串行，§14.4 第 3 锁）
  idempotent: bool,       // 并行 FC 前提（§14.4 三前提）
  latency_budget_ms: int  // 超时熔断阈值（§14.4 第 2 锁）
}
```

三域共存调度演示（"导航去 X、放首歌、调低空调"）：① Router 落域 → `multi_agent` skill，命中三能力；② 三者无数据依赖 → DAG 并行；但 car_control 与潜在车窗共 `exclusive_bus=hvac` → 同总线串行；③ 各 mock handler 返回 → Joiner 按 weight（导航>娱乐）合成；④ 主频道语音播报导航，音乐/空调弱展示。

v0.1 已知浅点（留接口未展开，供后续）：难度感知动态编排（VAE 估算 → 算子分配）、Re-planner 偏差阈值、FC 延迟分层（车控<100ms → 多步推理）、长期人格记忆。

## §15 LoRA 工程化闭环 ⭐

> 衔接：v0.1 §10 Codex 铁律⑦只给了一句「few-shot/gold eval/微调三者数据合同同源，先写 100 条黄金样例」，v0.1 §11.2 提了「数据合同同源（五者格式统一）」，但**全篇没有任何"何时练 / 练什么 / 怎么练 / 怎么量化导出 / Day1 数据管线"的工程闭环**。本节是 v0.1 最薄弱处的补强，重点解决 **模糊说 → 跨域动作映射**。所有协议/字段沿用 v0.1 §4/§6 语义结构。

### 15.1 何时该上 LoRA（决策门，不要过早练）

LoRA 不是起手项，是 prompt 撞墙后的下一步。三道门全过才练：

| 门 | 判据 | 不过的动作 |
|---|---|---|
| Prompt 已验证 | 改 system prompt / 工具 description（v0.1 §4.1 descriptor）/ few-shot 后，模糊说准确率仍 <90%（v0.1 §11.2 意图准确率门），且瓶颈是"同一感受映射到多个域选错"而非"工具定义缺失" | 先补 descriptor 语义线索、enum、2-3 触发示例，不练 |
| 样本攒够 | "模糊说 + 跨域"样本 ≥ 300–500 条且带期望标签；单域空调类 ≥ 150 条 | 继续用 §15.5 trace 管线攒，不练 |
| 端侧约束逼迫 | 演示要离线、0.6B zero-shot 选域不稳、不能塞长 few-shot（端侧 token 预算紧）| 若可联网用大模型则未必需要 LoRA |

口诀：**标准说走规则（不练，对接 v0.1 §10 铁律②规则吃 80%）→ 模糊说/自由说 prompt 撞墙 → LoRA**。标准说（"打开空调"）永远白名单匹配，不进训练集。

### 15.2 LoRA 训练数据合同（五件套，落地 v0.1 §11.2「数据合同同源」）

一条样本 = 一个不可变记录，五字段缺一不可。**与 §15.5 trace、v0.1 §11.2 eval 用同一 schema**，避免三套口径。

```jsonc
{
  "input": "I'm a bit cold, no need to keep the air conditioner on.",
  "context": {"domain_hint":"airControl","zone":"主驾","speed_kmh":0,"prev_turn":null,"lang":"en"},
  "tools": ["set_air_conditioning","open_window","set_seat_heating", "..."],   // 本轮可见工具子集（v0.1 §10 铁律③ ≤10）
  "expected": {"tool":"close_ac","slots":{"instype":"close"},"label":"模糊说"},
  "error_tag": null
}
```

| 件 | 字段 | 取值/来源 | 作用 |
|---|---|---|---|
| ① 输入 | `input` | 用户原话（多语种）| 训练 X |
| ② 上下文 | `context` | 区域/车速/温度/上轮/语种（v0.1 §8 DialogueState 子集）| 跨域消歧依据 |
| ③ 工具定义 | `tools` | 本轮可见工具子集（v0.1 §4.1 schema + §6 取值范围）| 约束输出空间，防域漂移 |
| ④ 期望输出 | `expected` | tool + slots(JSON) + label（标准说/模糊说/自由说/场景触发说，= v0.1 §11.2 三档语料）| 训练 Y |
| ⑤ 错误标签 | `error_tag` | 失败 case 才有：枚举（见下）| 难例挖掘 + 加权 |

`error_tag` 枚举：`域选错(跨域映射) / 槽缺失 / 槽越界(超 enum) / 口径漂移 / 语种串台 / 误吸(闲聊当车控) / 漏召(车控当闲聊) / 接口超时`。前 6 类是 LoRA 可学的；接口超时排除出训练集只进稳定性池。

> 合同硬约束：slots 必须落在 v0.1 §4.1/§6 取值范围内（如 `position` ∈ 主驾/副驾/全车…）；越界样本进 `槽越界` error_tag 而非静默丢弃。

### 15.3 MAformac 该练什么（重点：模糊说 → 跨域动作映射）

按 ROI 排序，演示助手只练前两类即够：

1. **模糊说跨域映射（P0，核心）**：一句感受 → 选对域 + 对动作。难点是同一感受跨多域（对接 v0.1 §11.1 金句表）：
   - "有点冷" → 空调升温 vs 座椅加热 vs 关窗 → 靠 context（座椅状态/车速/当前温度）消歧
   - "太吵了" → 关窗 vs 降音量 vs 降风量 → 跨 carControl/媒体/airControl
   - "太闷了" → 内外循环 vs 开窗
2. **自由说归一（P1）**："帮我弄一下温度" → `set_air_conditioning(temperature:23)`；动词模糊、参数靠推理。
3. **不练**：标准说（规则）、场景触发说（属 MasterAgent 阶段，端侧 0.6B 不碰）、多意图（产品约束不支持）。

数据配比（解决类别不均衡）：模糊跨域 50% / 自由说 25% / 标准说负样本（教模型"这条走规则别乱选"）15% / 闲聊负样本（压误吸，对接 v0.1 §11.4 拒识 ≥80%）10%。多语种（中/英/俄）分层抽样——**语种串台**是 v0.1 §11.4 实测高频失败（俄语问→中文答），需专门加样本。

### 15.4 评估（base vs LoRA 对比，整句帧准确率）

沿用 v0.1 §11.2 评分三层（意图→槽 F1→整句帧）+ §11.4 badcase 分布，落到 FC 微调对照：

- **整句帧准确率（主指标）**：tool 名 + 全部 slot 键值完全匹配（AST 精确匹配，BFCL 口径，呼应 v0.1 §10 repo④）。
- **分层指标**：域准确率 / 工具准确率 / 槽准确率 / **跨域子集准确率（最关键，单独看）**。
- **对照实验**：同一测试集跑 base(0.6B 裸) vs LoRA，出 4 列对照表，证明 LoRA 净增益且**误吸率不升**（≤5% 硬门，闲聊负样本集回归）。
- **硬门**（触发即 FAIL）：域选错跨域 / 槽越 enum / 语种串台 / 闲聊误吸。
- **三层回归集**：冒烟集 20–30（日跑）/ 标准集 80–120（每次训完）/ 高风险集 40–60（跨域+多语种+边界，导出前门禁，对接本篇 §16 must-pass）。

评分伪代码（端侧本地跑，不依赖大模型 judge）：

```text
frame_match(pred, exp) = (pred.tool==exp.tool) and (pred.slots==exp.slots)
hard_fail = cross_domain_wrong or slot_out_of_enum or lang_mismatch or false_trigger
acc_overall = sum(frame_match)/N ;  acc_cross = same over cross-domain subset
```

### 15.5 Day1 数据管线（交互 trace 落盘 = 训练集种子）

**最便宜且最关键的一步**：MAformac 第一天就让每次交互按 §15.2 五件套 schema 落盘，演示用着用着就攒出训练集。这是 v0.1 §10 repo⑦「记忆靠本地 JSONL event log」的训练用途延伸。

- **落盘点**：mock 车控执行前，把 `input / context / tools / 模型输出帧 / 命中规则还是模型` 写入本地 JSONL（端侧、离线、不上云）。
- **trace 字段对齐 eval 列**：直接复用 v0.1 §11.2 五件套表头当落盘 schema，零额外设计。
- **人审回填**：演示后挑 FAIL/可疑 trace，人工补 `expected` + `error_tag`（10 秒/条），晋级训练样本；PASS 高置信 trace 半自动入正样本。
- **难例优先**：带 `error_tag` 的（尤其跨域域选错）2–3 倍加权进下一轮。
- **隐私边界**：trace 只落 mock 场景，无真实车辆/PII（守边界声明）；演示数据可直接入仓，与生产数据规则隔离。

闭环：`Day1 trace 落盘 → 人审回填 → 攒够(§15.1 门) → LoRA 微调 → eval 对照(§15.4) → 难例反哺下一轮`。

### 15.6 训练流程（MLX-LM LoRA on Apple Silicon：数据→训练→量化 Q4→导出）

在 Mac 本机用 **MLX-LM**（Apple Silicon 原生，贴 v0.1 §10 repo① mlx-swift-lm 同生态），无需 GPU 服务器：

1. **数据准备**：五件套 JSONL → 渲染成对话/工具调用格式（`{messages:[{role:user, content:input+tools+context},{role:assistant, content:期望帧}]}`），切 `train.jsonl / valid.jsonl`（≈9:1）。工具定义放 system 段，期望帧（tool+slots JSON）作 assistant 目标。
2. **训练**：`mlx_lm.lora --model Qwen/Qwen3-0.6B --train --data ./data --adapter-path ./adapters --iters … --batch-size … --num-layers …`（LoRA 只训低秩矩阵，0.6B 单机分钟级–小时级）。`temperature=0` 推理；监控 valid loss + 整句帧准确率早停。
3. **融合（可选）**：`mlx_lm.fuse --model … --adapter-path ./adapters` 把 adapter 合进基座得单一权重，便于端侧分发。
4. **量化 Q4**：`mlx_lm.convert --hf-path <fused> -q --q-bits 4` → Q4，体积/内存大降，匹配 v0.1 §11.4 资源占用门（内存 ≤2.5GB / 磁盘 ≤8G）。
5. **导出 + 端侧验证**：导出 MLX 格式，集成进 MAformac（MLX Swift 端侧推理）；用 §15.4 高风险集做导出门禁，过线才算交付。**量化后必须重跑整句帧准确率**，确认 Q4 掉点在阈内（跨域子集最敏感）。

> 全闭环串联：§15.1 决策门 → §15.5 Day1 trace → §15.2 五件套 → §15.6 训练量化 → §15.4 base/LoRA 对照 → 难例反哺。这条线让 MAformac 的"模糊说跨域"能力可迭代、可量化、可端侧离线交付。

## §16 安全门控深化 + 必过集验收基线

> 衔接：v0.1 §4 把 L4 安全层定为"代码硬护栏（取值范围/反向护栏/ADAS拒识/确认门）"并给了错误枚举（v0.1 §4.2/§12 D3/D5），但**没有风险分级体系、没有确认/拒识的具体策略、没有 must-pass set 的定义与放行门级**。本节深化安全门控并补必过集验收基线，**不重复** v0.1 §11.4 demo 验收指标表（那是 KPI 数值，这里是门级与流程）。

### 16.1 安全门控完整分级（五门顺序 + R0–R3 + 反向护栏）

**五道门固定顺序**（绝不能先看偏好后看安全；是 v0.1 §4 L4「确认门」的展开）：

`语义门 → 对象门 → 风险门 → 权限门 → 环境门`

- 语义门：是否有明确车控意图（输出：意图类型/对象/参数/置信度）
- 对象门：控制对象是否清晰（主驾/副驾/后排/全车，对接 v0.1 §8 speaker_zone）→ 不明升级"待澄清"
- 风险门：高风险/不可逆/驾驶敏感 → 确认或拒绝
- 权限门：主账号/家庭/访客资格 → 执行或降级
- 环境门：当前车态（驾驶中/驻车/断网/低电）→ 执行或延后

**风险四级表（R0–R3，v0.1 仅笼统说"ADAS 拒识"，本表给完整分级）**：

| 级别 | 定义 | 示例 | 默认出口 |
|---|---|---|---|
| R0 低 | 可逆、低副作用、对象明确 | 调温、风量、阅读灯 | 直接执行 |
| R1 中 | 可逆但影响驾驶专注 | 切驾驶模式、多项联动 | 必要时确认 |
| R2 高 | 涉开口/车外/误触代价高 | 开车窗、天窗、解锁部分门 | 强确认 |
| R3 禁直达 | 驾驶敏感或不可逆 | 后备箱、儿童锁、远程放电、行驶中开后备箱 | 拒绝或延后 |

ADAS（v0.1 §5.1 lcc/lka/aeb/noa/tsr/盲区/碰撞预警）= R3，对应 v0.1 §4.1 `executable:false`。

**反向护栏（核心约束，v0.1 未点明）**：L3 偏好与 L2 任务规则**都不能把 R2/R3 降格成自动执行**——偏好只能改默认值，不能改风控级别。对象不明视同风险升级。确认**超时一律取消**（禁止"沉默即同意"，对接 v0.1 §9.2 不卡 UI）。

### 16.2 必须二次确认 / 必须拒识的动作

- **必须二次确认**：R2 全部（车窗/天窗/局部解锁）；对象不明的任何调节；写长期偏好（L3）；访客/副驾触发敏感动作。
- **必须拒识（拒绝或延后）**：R3 全部；视线安全类（雨刮/除雾/亮度，呼应 v0.1 §11.1「前挡起雾」金句）不可降成普通体验门；ADAS/驾驶态敏感动作行驶中禁直达。
- **隐私/多用户护栏**（深化 v0.1 §8 DialogueState 长期/短时分离）：访客模式默认不写 L3；切账号清 L1/L2 重载 L3；家庭账号 ≠ 主账号克隆（座椅/温度可共享默认值，行程/历史不互通）；第三人称、跨座位不默认执行，必过对象门（对接 v0.1 §8.2 hard gate「音区安全」）。

### 16.3 必过集（must-pass set）：是什么 / 怎么定 / demo 多少条

**定义**：不是"越多越好"的样例列表，而是**放量门槛 / release gate**——"绝不能错什么"的可回归门禁集。三集分账（v0.1 §11.2 把 eval 分了语料桶，但没分"演示/必过/量产"三集职责）：

- 演示集：证明亮点，讲解性强（= v0.1 §11.1 金句的执行载体）
- 必过集：控制主链路，任何回归不能掉
- 量产回归集：必过集 + Badcase（v0.1 §11.3）+ 长尾（demo 不需）

**怎么定**——按 **域 × 意图 × 风险**而非按功能。五大高频感知域：温度、声音、气味、视线、空间（= v0.1 §5.1 P0 四垂域 + 视线，+导航/搜索联动）。入集标准：高频真实表达 / 错了用户明显感知 / 能代表一类规则（非孤立句）。

**demo 该有多少条**：建议 **15–25 条精选必过集**，每域 3–5 条，覆盖句式（自由说/半自由说/第三人称/多音区/并列）。每条 case 字段：
`Query / 域 / 预期落域(多阶|FC|语义) / 推荐动作 / 端状态前提 / 风险级别(R0–R3)`。

### 16.4 验收基线 gate（不过不能演示）

**四层验收门**（全过才算通过，不能只看"模型大致理解"）：

`对象门(调谁) → 动作门(做什么、是否漏项) → 状态门(当前状态支不支持、已满足要扣减) → 风险门(能否直接放动作)`

> 状态门直接复用 v0.1 §7.2 三态判定（满足/部分/未满足/无法满足）；"已满足要扣减"= v0.1 §7.2「满足项不动」。

**放行模板（四态）**：

| 态 | 含义 | 可否演示 |
|---|---|---|
| `clear` | P0 必过样例全稳过 | 可演示/承诺 |
| `clear-with-limits` | 主线过，半自由说/候选有限制 | 可演示，标注边界 |
| `hold` | 对象门/状态门/风险门未收口 | **不能演示主承诺** |
| `candidate-only` | 仅演示，不进主承诺 | 仅演示 |

**放量门级**：P0 必过门（错一次回退）/ P1 强约束门（候选，需回退路径）/ P2 候选门（不写量产）。验收顺序即产品策略：**自由说达标后才放开半自由说**。

核心铁律（v0.1 未明说，最易踩）：**任何不能稳定复述的验收规则都不是规则；推荐动作完整性必须单列，不能藏在"总体效果"里。**

### 16.5 MAformac 演示版该保留哪些安全门（demo 也要安全感）

纯端侧 + mock 车控，demo 不真控车，但**安全门是讲解卖点**，建议保留 8 项：

1. **五门顺序可视化** — UI 显式跑「语义→对象→风险→权限→环境」，让观众看见拦截逻辑（最具说服力的 demo 元素，与本篇 §13.4 慢思考可视化同台展示）。
2. **R0–R3 风险标签** — mock 动作带风险色标；R2 弹二次确认卡、R3 直接拒绝并给替代路径（"停车后再试"）。
3. **状态机四出口** — `可执行 / 待确认 / 已拒绝 / 待澄清`（与 v0.1 §4.2 L4 错误枚举互补：那是失败码，这是放行态；量产再扩 已降级/已取消/超时）。
4. **优先级死规则**（伪代码顺序，本地 0.6B 不临场发挥）：`不清→澄清 → 违反安全→拒绝/确认 → 显式指令→执行 → L2→执行 → L3→建议默认 → fallback`。
5. **驾驶态输出分档** — 行驶中"一句话结论 + 一个动作状态"，驻车允许详细解释（接 v0.1 §5.3 VUI 只播操作对象）。
6. **解释边界** — 只说"为什么执行/拒绝/确认"，不暴露内部评分/规则权重/完整推理链（与本篇 §13.3 CoT 节流一致）。
7. **记忆三层 + 删除能力** — L1 会话/L2 任务/L3 偏好（映射 v0.1 §8 短时/长期分离），每层独立写读删；L3 带「来源证据」字段（显式说过 vs 连续观察）+ 账号归属；隐式偏好先入候选池累计阈值再问，不一次升 L3；可停用/清理/绕过。
8. **运行时埋点（mock 也展示）**：`risk_level / confirm_reason / clarify_reason / memory_source_used / account_scope / execution_result_state(success|reject|cancel|unknown)`。**「结果未知」绝不报成功**——demo 也演这条（强化 v0.1 §10 铁律⑤「读回 mock 态为准」），体现可追责。

一句话定位（demo 安全感的四条反向护栏）：会理解 ≠ 能越权；会学习 ≠ 能覆盖当前指令；会解释 ≠ 暴露推理；能推荐 ≠ 替用户做主。
