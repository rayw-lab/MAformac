## 1. 安全环节披露要求

### 1.1 为什么必须披露

当前 demo 已明确**不演任何安全拦截**；仍必须披露所有车控状态来自 mock state store、未接真车信号。否则观众会把 UI 读回误认为真车控制，形成安慰剂效应。

宪法红线：**mock 不得冒充真功能**（见 §4）。

### 1.2 口播脚本（开场 45 秒内，必须完整念完）

> 在正式演示之前，我先把前提说清楚，各位后面看 demo 时以这段话为准：
>
> 今天给大家看的是**演示原型**，不是量产形态。产品当前正向目录=12 个评审入口：空调调温是 18–32 整数边界模板，其余为精确话术；App 仍未接模型，多意图未准入，actionDemoProven=0/120，验收体系还在建设中，一条 V-PASS 都没有。
>
> 你们在屏幕上看到的车窗、空调、尾门等状态都来自 **mock state store**，不是真车机回读；真实档位、车速、CAN 与量产安全均未接入。界面左上角常驻「**模拟演示 · 非量产 · MOCK**」水印，请以水印为准。
>
> 演示过程中只要我没有明确说「已接真车」，默认都是有限评审目录与 mock 状态。我现在开始。

### 1.3 UI 水印文案（演示全程常驻，不得隐藏）

- **主水印**（屏幕左上角，高对比度，全程可见）：
  > **模拟演示 · 非量产 · MOCK**

- 当前脚本不演安全拦截，因此不显示会暗示安全能力的次级拦截水印。

- **水印硬性要求**：
  - 字号不得小于正文 70%；颜色与背景对比度 ≥ 4.5:1；
  - 不得被任何弹窗、动画、卡片遮挡；
  - 录像、截图、投屏场景下水印必须完整保留，不得裁掉；
  - 如果演示场地有投影色差风险，备份方案：口播开始时手指水印位置，口播重申。

### 1.4 其他披露触点

- PPT 首页标题下方必须出现事实锚原文："产品当前正向目录=12 个评审入口（空调调温是 18–32 整数边界模板，其余为精确话术）/ App 仍未接模型 / 多意图未准入 / actionDemoProven=0/120"；
- 演示录像开头 5 秒字幕必须包含主水印文案；
- 对外发布的 demo 视频描述 / 推文 / 海报必须包含「模拟演示，非量产」字样。

## 2. 三类必然追问应答（逐字稿）

这三类问题**几乎必然出现**，原因是观众本能想用「换个场景试一下」来探底。应答原则：**不即兴、不现场试、不演示脚本外行为**；用标准答劝退，不跟观众赌「换个说法能不能命中」。

### 2.1 追问一：「停稳能开吗？」（车停稳了，能不能开窗/升窗）

**标准答（逐字念）**：

> 当前演示目录里有一条评审过的开窗话术，但它只在 mock state store 与静态演示上下文中运行，**不代表已经接入真实档位、车速或 CAN 信号**。
>
> 所以我可以按脚本演示那条精确话术的执行和读回，但不能回答「停稳才允许」或「行驶中会拒绝」；这两项真实安全判定尚未接入，也不在本次验收范围。
>
> 任何超出评审话术的现场变体都不即兴执行，避免把一次偶然命中包装成泛化或安全能力。

**配套动作**：只按 `demo-script-v1.md` step4 的精确开窗话术演示；不改写句子，不演行驶态/停稳态安全判断。

### 2.2 追问二：「那开窗呢？/ 那空调呢？/ 座椅呢？」（横向探其他功能）

**标准答（逐字念）**：

> 当前正向目录有 12 个评审入口，覆盖十族语料、尾门和旧空调入口；它们会从 App 文本入口走同一条产品 runner，并用 mock state store 读回。
>
> 这仍然是有限白名单，不是自然语言泛化，也不是真车控制。模型、真实语音输入、多意图和车机信号都没有接入。
>
> 我可以严格按演示脚本逐条运行这些评审话术；目录之外不现场即兴，系统也应 fail-closed。

**配套动作**：只运行 `demo-script-v1.md` step4 列出的逐字话术；不替换同义句，不把 UI mock 状态称为真车回读。

### 2.3 追问三：「换个说法」（不用白名单字面，换种表述看能不能识别）

**标准答（逐字念）**：

> 我不现场换说法试。
>
> 当前是 12 个评审入口与空调调温参数模板，没有 NLU、没有做泛化，**目录之外的说法应当拒识**。偶然命中不能证明系统「听懂人话」，拒识也不是故障，而是当前 fail-closed 边界。
>
> 支持哪些字面说法写在 `demo-script-v1.md` 与产品 E2E manifest 中，以磊哥亲灌句 + 每周录像为准，不以现场即兴试的结果为准。

**配套动作**：不要输入任何文本；如观众坚持要试，礼貌重复「今天不做即兴测试，按脚本走」，然后进入下一环节。

## 4. 宪法红线：mock 不得冒充真功能

这是本文件**不可妥协**的一条线，优先级高于任何演示效果、现场氛围、领导观感：

1. **UI 上 mock 产生的状态**（车窗动画、档位、车速、空调卡片、温度数字）必须有可见标识，让观众能区分「这是 mock」还是「这是真车机回读」。没有标识的 mock = 冒充。
2. **脚本预设的拦截**（安全、权限、车况）必须在口播中明示「这是脚本预设」，不得说「我们的系统会…」「我们的安全逻辑会…」。v1 起本阶段**不再演任何安全拦截**，从源头消除冒充风险。
3. **未实现的能力**（云端/端侧模型、真实语音输入、自然语言泛化、多意图、记忆、白名单外其他车控功能）一律不演示、不暗示、不「以后会有」式画饼——画饼也是冒充（冒充未来确定会有）。白名单外输入一律说「没接 / coming soon」，不得包装成「安全机制」。
4. **验收状态**（actionDemoProven=0/120、App 仍未接模型、多意图未准入）不得用任何方式弱化、粉饰、换词；「建设中」「在路上」「推进中」等模糊语态若不带事实锚，按违规处理。
5. **红线违反 = 当场叫停**。在场任何一位发现演示者违反本条，有权当场打断并要求按本文件话术纠正；演示者不得抗辩。
