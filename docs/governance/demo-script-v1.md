## 1. 前置检查清单（演示前 5 分钟必做，缺一不开演）

| 项 | 检查内容 | 通过判据 |
|---|---|---|
| 1.1 数据流可见性 | 确认状态卡、对话区和隐藏 proof 文本均可见 | 输入后能读取 state key/value、revision、contract row 与 readback；不检查语音 |
| 1.2 Debug build | 确认当前运行的是最新 Debug build，水印「模拟演示 · 非量产 · MOCK」常驻左上角 | 启动后水印可见、未被设置按钮或任何卡片遮挡；字号 ≥ 正文 70% |
| 1.3 cold-start | 启动后无预置用户话术，只有中性的「我在听…」助手提示 | 对话区不存在伪造的历史用户操作；状态卡使用当前 contract defaults |
| 1.4 无 mock 行驶态 | 不触发任何模拟行驶/车门/档位开关（本版本不演安全拦截） | 确认无行驶态浮标、无车门状态指示器亮起 |
| 1.5 网络开关就位 | 准备好 Wi-Fi 开关 / 拔网口位置，step5 要在 10 秒内完成断网 | 手能直接碰到开关；提前想好断网动作，不临场找 |
| 1.6 计时 | 开计时器；**整场不超过 6 分钟**（开场 + 空调三步 + 目录外拒绝 + 断网 + 收尾） | 超时 = 压缩 Q&A，不删评审话术、不即兴加戏 |
| 1.7 口径自检 | 走 §6 禁说词 checklist，确保口播与字幕稿无禁说词残留 | checklist 全项打勾 |
| 1.8 candidate 禁演 | 确认演示者不会输入车窗 / 氛围灯 / 座椅 candidate literal | 脚本外 candidate 路径一律不演；即使代码存在也不得展示 |

## 2. 修订后 7 步演示脚本（可演示面=空调两族；整场录像待人验）

> 全程话术必须逐字念；文本框输入严格按引号内字面；**不做任何「换个说法试试」**；**不得演示后三族 candidate**。

### step0 —— 开场披露（约 45 秒，必须放在第一步）

1. 站定、手指屏幕左上角水印「**模拟演示 · 非量产 · MOCK**」；
2. 逐字念 `docs/governance/safety-disclosure-and-qa.md` §1.2 口播脚本（事实锚 + mock 标识 + 水印声明，约 45 秒）；
3. 念完再点一次水印，说「请各位后面看到的所有 UI 反馈都以这个水印为准」，再进入 step1。

### step1 —— 规则快路径：打开空调（≈ 30 秒）

| 维度 | 内容 |
|---|---|
| **文本框输入** | 「打开空调」 |
| **系统响应（预期）** | 空调卡片亮起，对话区显示来自 readback payload 的「空调已打开」 |
| **话术（逐字）** | 「第一步：规则快路径，命中白名单 literal → 执行 + 读回。现在空调是开的状态。」 |
| **动作** | 输入后停顿 2 秒，让观众看到卡片变化与 readback 文本，再说话术 |

### step2 —— 状态写入 + 读回确认：设温 26 度（≈ 40 秒）

| 维度 | 内容 |
|---|---|
| **文本框输入** | 「把空调调到26度」 |
| **系统响应（预期）** | 主驾温度数字变 26，对话区显示来自 readback payload 的「主驾空调温度26度」 |
| **话术（逐字）** | 「第二步：参数写入后读回确认。UI 上的 26 度和对话区读回来自同一份 state 数据。」 |
| **动作** | 手指温度数字停 1 秒，确认观众看到变化 |

### step3 —— 状态感知拒重复：已开不重复执行（≈ 30 秒）

| 维度 | 内容 |
|---|---|
| **文本框输入** | 「打开空调」（与 step1 完全相同字面） |
| **系统响应（预期）** | 不重复触发开动作；状态、revision 与 runner 次数均不增加，界面保留当前态 |
| **话术（逐字）** | 「第三步：状态感知的重复拒绝。系统知道当前状态已经是开，不重复执行。这是规则路径的状态判断，不是智能识别。」 |
| **动作** | 输入后停 2 秒；强调「同一个句子，第二次没有重复写入」 |

### step4 —— 诚实拒绝：目录外 / candidate 禁演（约 30 秒）

| 维度 | 内容 |
|---|---|
| **文本框输入** | 「打开车窗」 |
| **系统响应（预期）** | 无状态变更、无执行读回、runner 不调用；提示暂不支持 |
| **话术（逐字）** | 「这句不在当前可演示目录。车窗、氛围灯、座椅虽在研发候选里，但未发布、未验收，今天不演。系统没有执行——不是安全拦截，也不代表这些能力已就绪。」 |
| **动作** | 输入后核对没有卡片或状态值变化；只展示拒绝提示，不把拒绝包装成安全能力，**不改用 candidate literal（如『把主驾车窗再开50%』）试演** |

### step5 —— 断网重跑：全本地、零云依赖（≈ 60 秒）

| 维度 | 内容 |
|---|---|
| **动作** | 1. 当场关闭 Wi-Fi / 拔掉网线；<br>2. 等待 10 秒；<br>3. 关闭并重启 App，恢复默认 state store；<br>4. 依次重跑 step1「打开空调」与 step2「把空调调到26度」 |
| **系统响应（预期）** | 无网环境下两句响应与联网时一致：状态变更、readback payload 与 UI 同步 |
| **话术（逐字）** | 「现在断网重跑两个空调目录入口。行为没有变化，因为当前链路是本地规则、状态存储与 readback 数据流，没有云端模型兜底。」 |
| **动作** | 断网动作让观众看见；跑完后恢复联网 |

### step6 —— 收尾声明（≈ 20 秒）

逐字念：
> 最后再重申：今天看到的所有车控反馈都是**演示原型**，来自 mock state store 与**有限空调可演示目录**；车窗、氛围灯、座椅属于未发布 candidate，今天未演、未验收。它们不代表真车接入、自然语言泛化、多意图或量产安全。App 未接模型，`actionDemoProven=0/120`。验收以磊哥亲灌句和每周录像为准，今天的演示不构成验收信号。

## 5. 严格禁止项（演示者违约 = 当场叫停）

依据 `docs/governance/safety-disclosure-and-qa.md` §4 宪法红线与 `docs/governance/demo-messaging-guidelines.md` §1 禁说清单：

1. **不得即兴输入脚本外句子**（含「那我换个说法试试」「我随便试一句」——不管观众怎么鼓动）。
2. **不得演示后三族 candidate**（车窗 / 氛围灯 / 座椅 literal，含 `把主驾车窗再开50%`、`打开氛围灯`、`打开副驾座椅加热` 等）——代码存在≠可演示。
3. **不得触发任何 mock 行驶态 / 车门 / 档位开关**——本版本不演安全拦截，不演就不冒充。
4. **不得把 step4 的 coming soon 拒绝说成「安全机制」/「保护逻辑」/「功能安全」**——它只是「未发布 / 禁演」。
5. **不得念任何 §6 清单中的禁说词**；若观众使用这些词，用 demo-messaging-guidelines §3 标准答纠正，不附和。
6. **不得跳过 step0 开场披露**，哪怕时间紧——超时砍 Q&A，不砍披露。
7. **不得在断网前提前关闭网络**（破坏 step5 的对比效果），也不得跑完忘记恢复。
